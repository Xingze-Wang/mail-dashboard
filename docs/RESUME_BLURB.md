# Qiji Pipeline — Resume / Portfolio Blurb

## Version 1 — One-line headline

Built an AI-powered sales pipeline for venture-fund researchers, combining real-time lead scoring, prompt drift detection, and WeChat conversion tracking — deployed on Vercel + Supabase, powering Qiji Compute outreach to top Chinese AI researchers.

## Version 2 — Standard resume bullets

- Engineered a dual-target logistic regression model (10 features: local score, log-citations, school tier dummies, overseas/.cn signal, per-rep one-hots) trained on ~1,000 samples with sample weighting (4× for WeChat conversions, 1× for clicks) to predict researcher conversion; deployed on-demand via API with live inference at send time.
- Designed a daily weekday cron pipeline (sync from Resend → scan arXiv → enrich via Semantic Scholar → classify → assign → draft → drift-mine → emit retrain signals) that processes ~10,000 emails and ~300 new papers in under five minutes, with graceful degradation on enrichment failure.
- Built a drift-mining LLM pipeline that detects recurring patterns in sales edits to AI-generated drafts; produces structured JSON (category, AI phrase, sales rewording, occurrence count, suggested prompt patches) using Claude Opus with Gemini fallback; admin reviews and promotes patterns into the prompt.
- Implemented per-rep attribution and analytics with multi-dimensional drill-down (school tier, research direction, channel); computed lift metrics vs org baseline and surfaced segments only above a sample-size threshold to suppress noise.
- Architected a helper bot ("老师傅") with evidence-first design: the LLM proposes database lookups, the runtime executes them with per-rep scoping enforced server-side, and only then does the LLM answer — every claim is grounded in fetched data.
- Built hot role re-reads into auth: JWTs carry rep id and role, but role is re-fetched from Postgres on every request so demotions take effect instantly; system fails closed on DB errors rather than leaking access.
- Shipped end-to-end WeChat conversion loop: in-app "added on WeChat" button → `brief_lookups` row attributed to the actor (not the lead owner) → counted in training set → 4× weighted in next retrain → drives admin-facing retrain proposals.

## Version 3 — Long-form portfolio paragraph

I built and operate a production sales-intelligence platform for Qiji Compute (奇绩算力), a venture accelerator that identifies and supports top Chinese AI researchers with a free GPU compute grant. The system runs a unified daily pipeline on a Vercel Pro cron: it syncs roughly ten thousand emails from Resend, scans arXiv for new papers in compute-relevant fields, enriches each lead with Semantic Scholar (h-index, citation count, school tier), classifies the lead as "strong" or "normal," routes it to the right sales rep via a configurable rule set (strong tier, by-direction, overseas vs `.cn`), and generates a personalized email draft carrying that rep's voice and WeChat ID. The whole pipeline completes inside Vercel's 300-second cap with graceful fallbacks at every step — if Semantic Scholar times out, the lead still gets drafted; if drafting fails, the lead lands as `new` for manual handling.

The most interesting technical piece is a dual-target logistic regression conversion model. Email ground-truth signal is rare — we get a few WeChat additions per week — so I trained a single LR over both clicks and WeChat-adds with sample weights (4× for WeChat-positive, 1× for click-only, 1× for negative), so the model can learn from the high-volume proxy signal while still respecting that WeChat-add is meaningfully stronger per event. The feature vector is deliberately small (10 dimensions) to avoid overfitting on a thousand samples. The trainer pulls the email send log, unions `webhook_events` with `emails.status` so clicked-then-complained recipients aren't undercounted, applies a deterministic shuffle, and reports held-out AUC plus label composition so admin can read the metric in context rather than treating F1 as gospel.

A second LLM pipeline mines prompt drift. Each day, the cron feeds the previous day's edits — original AI HTML, sales-edited version, edit-reason tags, free-text notes — to Claude Opus, which extracts recurring patterns: phrases the AI overuses, formats that get rewritten, tone mismatches per rep. The patterns land in `prompt_drift_patterns` with status `pending`, the team reviews them at `/drift`, and accepted patterns are merged into the generation prompt. The loop closes itself — sales behavior measurably re-shapes the model that drafts for them.

The helper bot ("老师傅" — old craftsman) was designed against the failure mode of LLMs confidently inventing data. Every chat turn first runs a `get_my_stats` or `lookup_lead` tool with hard scoping (sales reps see only their leads — admins can read everyone), and only then does the model answer. No claim leaves the model without evidence in the same context window.

Per-rep attribution required a real architectural distinction I learned the hard way: the *actor* on a record (who clicked "Added on WeChat," who edited the draft) is not the same as the *owner* of the lead it touched. We separately store `marked_by_rep_id` on `brief_lookups` and `assigned_rep_id` on `pipeline_leads`, and conversion counts roll up by actor. Auth is paranoid by default: JWT payload carries rep id, but role is re-read from Postgres on every request so an admin demoted while logged in cannot continue viewing the team's leads. System fails closed on DB errors rather than ever leaking access.

The stack is Next.js 16 (App Router), React 19, TypeScript, Supabase Postgres (no RLS — app code enforces scoping with explicit per-route filters), Resend for outbound mail and webhooks, Anthropic Claude as the primary LLM with Gemini fallback. About 60 API routes, a dozen pages, and a custom UI built on shadcn primitives and Tailwind 4. The system is live with real sales reps, real researchers receiving outreach, and real WeChat conversions tracked daily.
